# Example 03 · Evaluator-optimizer, self-improving (TypeScript)

> **Synthetic / teaching example.** Runs offline against the deterministic mock provider — no API key,
> no network. The numbers below are generated by the run and checked against `receipt.json`.

The **evaluator-optimizer** pattern: **generate → critique-against-a-rubric → revise**, looping until a
rubric gate is met. One task here — write a product-page headline for an API observability tool. Each
iteration runs a generator (or reviser) LLM call to produce a draft, then an evaluator LLM call that
scores the draft 0–1 against a rubric and returns a one-line critique. If the score is below the gate the
critique is fed back into the reviser and the loop runs again (capped at 4 iterations). It owns the
**evaluator-optimizer** pattern from [the pattern catalog](../../../docs/03-pattern-catalog.md).

## Run it

```bash
npm install
npm run run     # writes trace.jsonl, eval-report.json, receipt.json
npm test        # Vitest
```

Re-running is a no-op diff: the run is deterministic (canned fixtures, a synthetic clock — no `now()`,
no RNG).

## The receipt

The loop took **3** iterations. The rubric score climbs **0.55 -> 0.72 -> 0.88** and crosses the gate of
**0.85** on the third pass, so the final score is **0.88** and the receipt records `gatePassed: True`.
The run emits **9** trace events conforming to
[`observability-event-schema.json`](../../../templates/observability-event-schema.json).

| Artifact | What it holds |
|----------|---------------|
| [`trace.jsonl`](trace.jsonl) | every generate / evaluate / revise `llm_call` + each gate `decision`, one event per line |
| [`eval-report.json`](eval-report.json) | the per-iteration score-climb ledger (iteration, score, passed) |
| [`receipt.json`](receipt.json) | the headline numbers (machine-checked against this README) |

## How it works

1. `eval/task.json` is the task: the brief, the rubric, the gate (`0.85`), the iteration cap, and the
   recorded score-climb script.
2. `record-fixtures.ts` (off in CI) records the canned drafts + evaluator scores, so the run is offline
   and the eval tests the **loop logic** (does the gate stop the loop?), not the model.
3. `run.ts` runs generate → evaluate → revise until the gate is met, then writes the receipts.
4. An unrecorded task throws `MissingFixtureError` — a drifted prompt fails loudly rather than silently
   hitting a real API.

The score-climb ledger is the point: a self-improving loop earns its keep only if the evaluator's score
**actually climbs** and the gate **actually stops** the spend. The tests assert both — the final score is
`>=` the gate and the score is monotonically non-decreasing across iterations.

See [chapter 02 — decision framework](../../../docs/02-decision-framework.md): this is a bounded
**workflow** (a fixed loop with a gate and a hard iteration cap), not an open-ended agent.
