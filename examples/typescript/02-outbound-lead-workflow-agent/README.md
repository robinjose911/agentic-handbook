A **durable-style outer workflow** wrapping an **inner agent loop**, with two governance guardrails the
raw loop lacks: a **HITL approval gate** on the one side-effecting action, and a **pre-execution
per-lead budget cap**. It owns the **workflow â†’ human-in-the-loop â†’ cost-control** patterns from
[the pattern catalog](../../../docs/03-pattern-catalog.md).

Per lead (from `eval/leads.json`, 5 leads), the inner loop runs:

```
enrich (LLM call) --> score (LLM call) --> decide next action --> [HITL gate if "send outreach"]
```

- **`send_outreach`** is side-effecting, so it cannot fire on the agent's say-so â€” it parks in
  `awaiting_approval` until a human approves (or is `rejected` if they decline).
- **`nurture` / `disqualify`** are non-side-effecting, so they auto-resolve with no human.
- **Budget cap.** Each step is bounded by `max_tokens`, so its cost can never exceed a per-step
  ceiling. Before *every* step the workflow **reserves that ceiling** against the cap; if the
  reservation would breach the cap, the step is **refused and never sent to the provider** â€” the lead
  stops with `budget_exhausted` (logged, not executed). A runtime assert enforces that a step's actual
  cost never exceeds the ceiling, so a single over-ceiling step can't slip through. One lead (`l3`, a
  long RFP dump) produces a costly `enrich`, so reserving the ceiling for `score` would breach the cap
  and that step is refused pre-execution.

Every decision is logged with its reason.


## The receipt

The workflow processes **5** leads and the **budget cap is honored** for every one: the per-lead cap is
**0.015** USD, and the highest actual per-lead spend (`maxLeadCostUsd`) is **0.006** USD â€” under the
cap, so the `budgetHonored` flag in the receipt is **True**. The run emits **16** trace events conforming to
[`observability-event-schema.json`](../../../templates/observability-event-schema.json).

Across the 5 leads: 1 approved (queued to send), 1 rejected by the human, 1 auto-nurtured, 1
auto-disqualified, and 1 budget-stopped.

## How it works

1. `eval/leads.json` is the leads set (5 leads â†’ expected decision + a canned HITL outcome).
2. `record-fixtures.ts` (off in CI) records the canned enrichments/scores **and their token usage**, so
   the run is offline and the cost model is deterministic. Lead `l3`'s deliberately fat `enrich` usage is
   what drives it over the cap at `score`.
3. `run.ts` runs the outer workflow over every lead, scores each decision against its label, and writes
   the receipts. It exits non-zero if the cap is ever breached.
4. An unrecorded lead throws `MissingFixtureError` â€” a drifted prompt fails loudly rather than silently
   hitting a real API.

The cost cap, the HITL gate, and the labelled illustrative pricing all live in `src/cost.ts` and
`src/agent.ts`. See [chapter 02 â€” decision framework](../../../docs/02-decision-framework.md): this is a
**governed workflow** (durable outer loop, gated side effects, pre-execution cost ceiling), not an
open-ended agent given a budget and trusted to stay inside it.
