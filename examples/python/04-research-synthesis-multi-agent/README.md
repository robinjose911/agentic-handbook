# Example 04 · Research-synthesis multi-agent (Python)

> **Synthetic / teaching example.** Runs offline against the deterministic mock provider — no API key,
> no network. The sources are a toy corpus and the numbers below are generated by the run and checked
> against `receipt.json`.

The pattern is **orchestrator-worker with explicit artifact handoff**. A research question fans out to
worker agents (one per mock search result); each **worker** extracts structured *claims* (one mock LLM
call). The **orchestrator** collects those claim artifacts, groups them into *themes*, and runs a
*synthesis* step that writes an answer **with a source reference per claim**. Finally a **groundedness
evaluator** checks that every synthesized claim cites a source that exists in the corpus. Roles are
separate and hand off **structured artifacts (claims, themes), never prose**. It owns the
**orchestrator-worker → grounding** pattern from [the pattern catalog](../../../docs/03-pattern-catalog.md).

## Run it

```bash
.venv/bin/python record_fixtures.py   # off in CI — authors the canned fixtures (only when prompts change)
.venv/bin/python run.py               # writes trace.jsonl, eval-report.json, receipt.json
.venv/bin/python -m pytest -q         # the test suite
```

Re-running `run.py` is a no-op diff: the run is deterministic (canned fixtures + a synthetic clock — no
`datetime.now()`, no RNG).

## The receipt

The groundedness score is **100%** — every one of the synthesized claims is grounded (**8/8** carry a
source reference that exists in the corpus), clearing the grounding gate of 1.0. Token usage is split by
role: **530** orchestrator tokens (the group + synthesis steps) and **520** worker tokens (the four
claim-extraction calls), for 1050 total. The run emits **7** trace events conforming to
[`observability-event-schema.json`](../../../templates/observability-event-schema.json).

| Artifact | What it holds |
|----------|---------------|
| [`trace.jsonl`](trace.jsonl) | every worker/orchestrator `llm_call` + the groundedness `decision`, one event per line, each LLM call tagged with its `role` and token usage |
| [`eval-report.json`](eval-report.json) | per-claim groundedness verdicts + the per-role token split |
| [`receipt.json`](receipt.json) | the headline numbers (machine-checked against this README) |

## How it works

1. `eval/golden.json` holds the research question, the toy search corpus (one source per worker), and
   the groundedness rubric (gate, expected themes, expected claim count).
2. `record_fixtures.py` (off in CI) records the canned worker claims, theme grouping, and synthesis —
   so the run is offline and the eval tests the **orchestration + grounding logic**, not the model.
3. `run.py` runs the pipeline: workers → claim artifacts → orchestrator groups themes → synthesis with
   source refs → groundedness eval, then writes the receipts. Token usage is summed per role.
4. An unrecorded request raises `MissingFixture` — a drifted prompt fails loudly rather than silently
   hitting a real API.

The handoff is the lesson: workers and the orchestrator are **separate roles** exchanging **typed
artifacts**, and a downstream evaluator enforces the **T**rust surface (every claim is grounded). See
[chapter 02 — decision framework](../../../docs/02-decision-framework.md): this is a **multi-agent
orchestration** (breadth-first fan-out + synthesize), the case where multi-agent earns its keep.
